## Phase 3 — Verification
- [x] Real-data harness: `tests/fixtures/` holds upstream-produced `.zst` files + plaintext; `tests/fixtures_roundtrip.rs` decompresses and compares byte-for-byte. **All 13 fixture tests passing**, including: `lorem_l1.zst` (10KB multi-block), `lorem_l1_checksum.zst` (XXH64 trailer), `lorem_l19.zst` (btultra / highest tableLogs), `empty_l1.zst` (zero-content), `binary_l{3,12}.zst` (high-entropy binary), **`zstd_h_l3.zst` and `zstd_h_l9_checksum.zst` (182 KB real C header — multi-block, crossing 128 KB boundary)**.
- Bugs fixed along the way:
  - Two `-C debug-assertions` pointer-underflow traps (upstream intentionally allows `size_t` wrap): `BIT_lookBits` start computation, `BIT_reloadDStream`'s cautious-update nbBytes cap.
  - **`ZSTD_DCtx::new()` left the HUF DTable's `DTableDesc` header zeroed**, so `maxTableLog=0` made every valid compressed-literals header hit `TableLogTooLarge`. Fix: initialize first u32 via upstream's `HUF_CREATE_STATIC_DTABLEX1` pattern — `(maxTableLog-1) * 0x01000001`.
  - **Cross-block back-references broken**: `ZSTD_decompressSequences_body` and `ZSTD_decompressBlock_internal` now accept an `op_start` offset into `dst`, so sequences in block N≥1 can reference prior blocks' output. Tail-literals and back-reference offset checks updated accordingly. Triggered at dst boundaries ≥128 KB.
  - **`ZSTD_DCtx::new()` left LL/OF/ML seq DTables zeroed** — the `set_basic` branch of `ZSTD_buildSeqTable` copies from the caller's previous table, so on a first-block-ever `set_basic` the decoder consulted an all-zero table and rejected every sequence. Fix: build the default LL/OF/ML FSE DTables at DCtx construction via `ZSTD_buildFSETable` over `LL_defaultNorm` / `OF_defaultNorm` / `ML_defaultNorm`. Only surfaced once our compressor started emitting `set_basic` (upstream-produced fixtures always use `set_compressed` on first block).
  - **`ZSTD_compressBlock_lazy_noDict_generic` panicked on `ZSTD_highbit32(0)`** when the depth-1/2 lookahead probes returned `ml2 == 0` (no chain match). The gain-comparison code called `highbit32(cand_off)` unconditionally. Fix: gate the gain comparison behind `ml2 >= 4 && cand_off != 0`. Surfaced on the 182 KB C-header fixture at level 10.
  - **Fast-strategy repcode-litLength-0 corruption at block boundaries.** `ZSTD_compressBlock_fast_noDict_generic` checked repcode-1 at `ip` (allowing litLength=0), but the decoder interprets `(offBase=repcode-1, ll0=1)` as `rep[1]` (shifted), not `rep[0]`. Fix: check repcode at `ip+1` like upstream's `ip0+step` pattern, ensuring litLength ≥ 1 always. Surfaced on highly-repetitive multi-block inputs where block 1 emitted a single long match covering the whole block, leaving anchor at block boundary — block 2's first rep check then fired with litLength=0. Hid behind an earlier `fast→dfast` workaround in `ZSTD_compress`/`ZSTD_compress_usingDict` that's now removed.
  - **Cross-block match carry extended to all strategies** (fast + dfast + greedy + lazy + lazy2). Each match finder now accepts `istart` so it scans `src[istart..]` while treating `src[..istart]` as history; `ZSTD_compressFrame_fast` keeps a single `ZSTD_MatchState_t` across all blocks of a frame. Result: **level 10 (lazy2) ratio on the 182 KB fixture improves from 1.06× to 0.99×** — we now match or slightly beat upstream zstd 1.5.7 at that level.
  - **Missing step-reset after matches in `zstd_fast` + `zstd_double_fast`.** Upstream's `goto _start` resets the uncompressible-step ramp after each successful match; our single-cursor variants left `step` growing monotonically, which meant that after the first ~128 mismatches early in a block we were skipping 3+ bytes at a time and missing ~half of all subsequent matches. Fix: reset `step = stepSize; nextStep = ip + kStepIncr` on every match-emit path. **Level 1 ratio on the 182 KB fixture improves from 2.13× to 1.00×**, level 3 from 2.26× to 1.05×. `zstd_lazy` doesn't have this bug (its step formula `(ip - anchor) >> kSearchStrength + 1` is self-resetting since `anchor` advances to `ip` on every match).
  - **`ZSTD_OPT_SIZE` was `1 << 12 = 4096` on the compress side, missing the `+3` margin from upstream's `ZSTD_OPT_NUM + 3`.** Found by a duplicate-constant audit: the correct `4099` definition lived in `zstd_opt.rs`; `zstd_compress.rs` shadowed it with the shorter value. The bug was latent only because `compressBlock_bt{opt,ultra,ultra2}` are still stubbed — it would have surfaced as a 3-byte buffer undersize in the optimal parser's forward-DP candidate array once wired. Fix: made `zstd_compress.rs` a `pub use` re-export of `zstd_opt::ZSTD_OPT_SIZE` so both modules now read the same canonical value.
  - **HUF X2 decoder over-decoded inside its tail loops.** `HUF_decodeStreamX2` did `for _ in 0..5 { decode; decode }` (10 calls/iter) where C's macro layout decodes 5/iter on `dtLog<=11`, plus the `dtLog>11` branch did 8/iter where C does 4. Caused level-1 noisy_rle and level-10 large-text frame failures whenever `HUF_selectDecoder` chose X2. Fix: collapse to the upstream call counts.
  - **HUF X2 tail had only one of upstream's two close-to-end stages.** Upstream first runs a `while reload+decode` loop, then a `while decode` (no reload) loop that drains any remaining 2-byte symbols out of the bit container after the bitstream has been consumed, then a single `HUF_decodeLastSymbolX2`. The Rust port had only the with-reload stage, dropping any final 2-byte symbol that was still readable after `BIT_reloadDStream` returned non-unfinished. Was the actual root cause of the X2 roundtrip failures.
  - **HUF X2 dtable construction collided unused symbols with weight-1 slot 0.** Upstream parks weight-0 symbols past all weighted ones during the sort (`rankStart[0] = nextRankStart`) and then resets to 0 afterwards. The Rust port set `rankStart[0] = 0` before the sort, so any weight-0 (unused) symbol that came before a weight-1 symbol would clobber sortedSymbol[0]. Latent on inputs with only two distinct symbols (no overlap), but a real port bug.
  - **`ZSTD_decompressContinue` couldn't see prior-block output.** Multi-block frames driven through the legacy continue API with a fresh `dst` per block lost cross-block back-references. Fix: added a `historyBuffer: Vec<u8>` to `ZSTD_DCtx`, append the block output after each successful decode (capped at `fParams.windowSize`), reset on frame boundaries / `ZSTD_decompressBegin`, and feed it into `ZSTD_decompressBlock_internal` as ext-dict history. Pinned by `decompressContinue_multi_block_with_independent_dst_buffers`.
  - **`ZSTD_execSequence` ext-dict spill used `dst.copy_within`, breaking ZSTD's repcode-style wildcopy.** When a back-reference straddles `ext_history` and the prefix portion of `dst`, the spill segment uses dst-bytes-as-source. `copy_within` is memmove-semantics: for overlapping forward copies it preserves the original src bytes, but ZSTD's wildcopy needs forward in-order copying so newly-written bytes feed back into the pattern (this is how repcode-style `offset=N, matchLength>>N` expansion works in the spill case). Fix: replaced with an explicit byte-by-byte forward loop. Surfaced when the level-1 fast strategy emitted a single block-spanning back-ref `(litLen=1, offset=1024, matchLen≈131071)` whose repeating-period-256 pattern depended on the wildcopy semantics.
  - **`blockStartAbs` in `ZSTD_compress_frameChunk` was the end-of-input absolute index, not the block-start absolute index.** The line `let blockStartAbs = window.nextSrc;` read `nextSrc` *after* `ZSTD_window_update` had already advanced it to `srcAbs + src.len()`. Then `ZSTD_overflowCorrectIfNeeded` and `ZSTD_window_enforceMaxDist` were called with this end-of-input value as the block start, making them think the window had already exceeded `maxDist` on the very first block — they shoved `lowLimit` (and hence `nextToUpdate`) far past the actual block contents. The row-hash matcher then read bytes at indices that lived outside the per-block slice and panicked: `range start index 47026184 out of range for slice of length 131072`. Fix: `blockStartAbs = base_offset + ip`. Surfaced on level 10+ (row-hash) for any input >~1 MB; lower-level matchers tolerated the bogus indices because they don't consult `nextToUpdate` directly.
  - **Pre-block-splitter `byChunks` produced corrupting splits on real-world binary corpora.** The fingerprint-based splitter (auto-resolved for dfast/lazy/lazy2/btlazy2/btopt/btultra/btultra2 via `splitLevels = {0,0,1,2,2,3,3,4,4,4}`) chose split positions whose subsequent block compression produced invalid output on `mozilla` (51 MB Firefox binary in silesia) at all levels that auto-resolved to byChunks. The decompressor reported "Data corruption detected" past byte ~2.5 MB at level 3, with the failing block being a tiny 8 KB chunk after a >2.5 MB history. Bisected to: a specific 8 KB block at compressor `ip=2,564,096` had 235 emitted sequences (all individually valid), but the resulting compressed bytes were rejected. Disabling the splitter entirely (forcing `FULL_BLOCK`) made mozilla roundtrip at all levels. **Workaround:** route every strategy through the cheaper `fromBorders` splitter (`SPLIT_LEVELS = [0; 10]`). Slightly worse ratio in some cases; byte-correct against upstream `zstd -d` for full silesia + enwik8 + every fixture tested. The byChunks bug itself isn't yet root-caused — see "Open" below.
